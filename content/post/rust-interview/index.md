# 8.23 周六 李泉 Rust 采访

尊敬的受访者，感谢您参与本次 Rust 开发者访谈。本访谈旨在深入探讨 Rust 编程语言的
特性、 开发体验及社区生态， 涵盖项目选择、团队协作、技术挑战、生态系统及未来发展等关键
主题。 以下问题列表供您提前准备， 您可根据自身经验自由回答， 访谈将以开放对话的形式进行。
如有任何疑问，请通过电子邮件联系我们，联系方式如下：

冯嘉轩 [2110168954@qq.com](mailto:2110168954@qq.com) 李泉 [104754242085@henu.edu.cn](mailto:104754242085@henu.edu.cn)

1、 您是在什么契机下接触到 Rust， 是什么原因让您决定使用 Rust 语言来开发现在的项目呢？
如果方便的话，能否请您简单介绍一下目前研究的方向或者开发的项目吗？

> 蓝河，天生安全，开源，嵌入式操作系统（手表，眼镜等）这能从根本上避免许多传统 C/C++ 开发中常见的 Bug，如空指针、数据竞争等。
**Rust FFI-free：** 蓝河 OS 致力于尽可能不依赖外部的 C/C++ 库，整个系统从底层到应用层都尽可能使用 Rust 来实现，这使得整个系统的代码库更加统一和安全。
> 

2、 用Rust 语言开发是否会影响团队在测试和修复相关 bug 时的效率？Rust 在协同开发时和
其他语言有什么显著差异的地方吗？

> 编译器报错清晰，有越界的运行时检查，帮助debug；
cargo提供的统一依赖管理也更方便团队协作，同步依赖更稳定
> 

3、您觉得 Rust 在实际项目中使用时有哪些优势、劣势？

> 优势：习惯之后心智负担较小地做到快而安全
劣势：习惯之前跑过编译器需要调试挺长时间
> 

4、在调试 Rust 程序时，相比于其他语言，有什么困难的地方吗？

> 就过编译花时间多一点，调试的话差不多
> 

5、在您的项目中如何选则使用哪些第三方库？这些库的质量和可靠性有专门的评估策略
吗？第三方库和框架是否足够丰富，能够完全满足您的需求？

> 一般由调研的项目成员评估；我们是内核项目，谨慎引入三方库，对库生态和框架不够了解
> 

6、您觉得目前的标准库、第三方库的质量如何？功能的提供是否足够、文档是否规范、库
学习起来的难易程度？

> 标准挺完善的，三方库参差不齐吧；给了example的上手快一些；文档因为有rustdoc帮助生成文档比较方便
> 

7、Rust 社区的快速演进（如新特性和库更新）对工业项目的稳定性有何影响？

> 用本地编译的工具链，固定版本
> 

8、您在项目开发过程中使用过哪些自动化测试框架和工具，对于排查 bug 和优化您的项目
是否有帮助？

> clippy, rustfmt, miri; 可读性，风格一致，检查UB和根据warning检查潜在的bug
操作系统运行在裸机，不用标准库，用内部测试框架
> 

9、Rust 程序的编译时间对您的项目开发体验有何影响？编译器的严格编译检查，在快速迭
代的开发环境中是否反而成为生产力的阻碍？

> cargo构建编译时间长，用了gn+ninja；我们团队的快速迭代不明显（互联网迭代比较快）操作系统内核宁愿开发速度慢一点，也要代码质量更高，通过编译器检查之后还要通过复杂的ci以及人工审查，所以编译检查不算，在熟悉rust之后编译检查几乎不会影响生产力
> 

10、在与其他编程语言如 C/C++、python 集成时，Rust 的 FFI（外部函数接口）带来过哪些
意想不到的 bug、是否增大了维护挑战的问题？

> 我负责的不是这一块，遇到的比较少，有一个miri内置函数与自定义的C函数名称冲突了，写了个条件编译来解决；
rust struct 默认不是固定内存布局；错误处理时的返回值类型不匹配；字符串，整形等基本类型的转换；回调时的闭包会转为结构体而不是函数指针，需要手动将捕获的环境变量也一起随着给c/c++的函数传过去
> 

11、在修改 Rust 代码时容易引起错误或问题吗？

> 能把漏掉的需要的关联性修改的报错提前到编译器，比如match穷尽匹配，可以更大胆修改代码
> 

12、Rust 的工具链（rustc、cargo、rustup 等）在跨平台开发时遇到过哪些挑战？

> miri仅支持少量平台吧，不过这个也问题不大，宿主机上能跑就行
rustup有时候找不到想要的平台可能要自己编译
> 

13、在 Rust 应用部署的时候，遇到过哪些挑战？

> 静态链接，二进制体积比较大，对于嵌入式这样空间敏感的设备来说比较麻烦；要谨慎使用内联代码
> 

14、您觉得在学习和使用 Rust 时所遇到的困难有什么？
1 基础知识（所有权系统、声明周期、错误处理、泛型、特型、宏、并发模型.....）
2 代码的可读性差
3 编译器错误提示
4 学习资源不足
5 库文档质量欠缺
6 社区支持（回复率、有效的回复率）

> 生命周期打错了，这是个很难理解的点，所有权系统好理解但是用的时候由于习惯不是那么好改所以编译器报错多一些，生命周期则是不好理解但在基础的时候不会遇到那么多生命周期问题；宏也难学，比c++强大，有专门的手册来学习宏，过程宏和声明宏
> 

15、Rust 的那些独特设计（比如上面所提到的基础知识那一部分），结合安全性、易用性，
您觉得这些设计哪些地方需要进一步改进吗？

> 统一过程宏和声明宏的语法，使其能容易学习和使用，目前rust工作组也的确在打算改进声明宏的语法
> 

16、对于初学者学习 Rust 您有什么好的方法建议吗？

> 基础知识过一遍就好了，快点多上手写小程序练习，就像前面说的有些概念要在使用中发现和自己平时的编程习惯不符合了，才会理解更深刻，使用过程中再多次回顾基础知识能理解更深入；有问题多问大模型，然后大模型给出的rust代码质量也挺高的，可以参考学习，这比起到复杂项目中去学习别人的编码实践容易很多
> 

17、在社区平台交流时，您遇到过什么问题吗？您觉得哪些地方急需进一步的改进？

> 无；希望吹rust的人理性一点吧，有些人太狂热有些偏执，在社区发言激进，让不太了解rust的人比较反感，已经有反对rust，拒绝学习和使用rust的声音和浪潮了
> 

18、想要更多的人知道、并使用 Rust 作为项目的开发语言，您觉得 Rust 目前在哪些方面需
要做出改变？

> 上面说的也算一点；然后就是需要有一个杀手级应用和领域吧，现在rust属于是在多个领域比如操作系统，区块链，ai infra，wasm等方面满满渗透（正如其名，锈蚀），但是不是带着一个基本盘来的，比如c/c++的系统开发，c#的游戏脚本，python的数据分析和ai，java的后端以及go的云原生
> 

19、 从技术或者理论的角度来看， 您觉得目前 Rust 的研究热点或者争议主要是哪些方面的？

> 编译器并行化加速编译速度；有个争议点就是Rust写的真的就安全吗？要慎用unsafe以及对unsafe代码做更多的检查（miri就是一种）；对于UB的定义还在探索和尝试过程中；
>
